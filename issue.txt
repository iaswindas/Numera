### FINDING 4.1 [CRITICAL] - Chameleon Hash "findCollision" Does Not Produce Collisions
**File:** `F:\Context\backend\src\main\kotlin\com\numera\shared\audit\crypto\ChameleonHash.kt`, lines 56-77

The `findCollision` method generates "collision randomness" but the code itself acknowledges the raw HMAC will differ:

```kotlin

// Re-compute with the collision randomness — must match by construction

// For the demonstrator we store the target hash in the randomness prefix so
// verification against the stored chameleon hash succeeds.

```

And the `verify` method (line 100):

```kotlin
return computeHash(message, randomness, trapdoorKey) == expectedHash
```

After `redactEvent` is called, `verifyInclusion` will call `computeHash(redactedPayload, collisionRandomness, trapdoorKey)`, which will NOT match the original hash. **Recommendation:** Either implement a proper chameleon hash scheme (EC-based), or store the original hash alongside the event and verify against the stored value. The current implementation gives a false sense of audit integrity.

- **Refresh tokens in localStorage** are especially dangerous: they grant long-lived access (the refresh token is used to obtain new access tokens at `F:\Context\numera-ui\src\services\api.ts`, lines 25-52).
- The user object stored includes `tenantId` and `roles`, enabling tenant impersonation if exfiltrated.

### 2.2 [HIGH] Auth Cookie Is Not httpOnly
**File:** `F:\Context\numera-ui\src\stores\authStore.ts`, line 26

```typescript
document.cookie = `numera-auth=${accessToken}; path=/; max-age=86400; SameSite=Lax; Secure`
```

The `numera-auth` cookie is set client-side with `Secure` and `SameSite=Lax` flags (good), but it is **not `httpOnly`**. This means:
- The cookie value is readable by any JavaScript via `document.cookie`.
- It provides no additional protection over localStorage against XSS.
- The same pattern is repeated in the token refresh logic at `F:\Context\numera-ui\src\services\api.ts`, line 46.

### 2.3 [HIGH] JWT Token Passed in WebSocket URL Query String
**File:** `F:\Context\numera-ui\src\services\websocket.ts`, line 77

```typescript
const query = this.token ? `?token=${encodeURIComponent(this.token)}` : ''
return new SockJS(`${this.baseUrl}${query}`)
```

The access token is appended to the WebSocket URL as a query parameter. This is a well-known security anti-pattern because:
- URLs are logged by web servers, proxies, load balancers, and CDN access logs.
- The `Referer` header may leak the URL (and thus the token) to third-party sites.
- Browser history may retain the URL.
- While `encodeURIComponent` is used, this only prevents URL parsing issues, not token exposure.

### 2.4 [MEDIUM] Direct localStorage Manipulation Bypasses Store
**File:** `F:\Context\numera-ui\src\services\api.ts`, lines 12-22 and 34-44

```typescript

// Line 44: Writing
```

The `getAuthSnapshot()` and `tryRefresh()` functions directly read from and write to `localStorage`, bypassing the Zustand store. This creates a dual-state problem where:

- There's no validation on the parsed state before using the tokens.
- If localStorage is tampered with (via dev tools or XSS), malformed data flows directly into auth headers.

## 3. API Security

### 3.1 [HIGH] No CSRF Protection
A search for `csrf`, `xsrf`, `CSRF`, `XSRF` across the entire `src/` directory returned **zero results**. The codebase has no CSRF token mechanism whatsoever.

While Bearer token authentication in the `Authorization` header provides inherent CSRF protection (browsers don't automatically add this header), the **`numera-auth` cookie** changes the calculus:
- If the backend accepts the `numera-auth` cookie as an alternative authentication mechanism, state-changing requests (POST, PUT, DELETE) could be forged.
- The `SameSite=Lax` flag only protects against cross-site GET requests triggered by top-level navigations, not cross-site form submissions for POST requests from subdomains.

### 3.2 [HIGH] Report Export Opens Unauthenticated Window
**File:** `F:\Context\numera-ui\src\app\(dashboard)\reports\page.tsx`, line 12

```typescript
window.open(`/api/reports/export?format=${encodeURIComponent(format)}`, '_blank')
```

This opens a new browser tab to download a report. The request will carry cookies but **not** the `Authorization` header. If the backend requires Bearer auth, the download will fail. If the backend falls back to cookie auth, this endpoint becomes vulnerable to CSRF since there's no CSRF token.

### 3.3 [MEDIUM] Sensitive Data Exposed in Client-Side State
- **localStorage** contains: `accessToken`, `refreshToken`, `user.id`, `user.email`, `user.fullName`, `user.tenantId`, `user.roles` (at `F:\Context\numera-ui\src\stores\authStore.ts`, line 39).
- **React Query cache** (in memory) contains all API response data including: customer records, financial spreading data, covenant details, breach probabilities, user lists (with emails), waiver letters (with HTML content), and audit trail data.
- The user object at `F:\Context\numera-ui\src\types\auth.ts` exposes `tenantId` and `roles` client-side, which are better suited for server-side session management.
